## 质量红线

### 测试覆盖率

- 核心业务逻辑测试覆盖率必须 **≥ [阈值，如 80]%**
- 所有 API 端点必须有对应的集成测试
- 数据库迁移脚本必须有回滚测试

### 代码质量

- 任何 PR / MR 必须通过所有自动化测试才能合并
- Lint 检查零警告（配置文件内明确忽略的除外）
- 安全相关代码（认证、授权、数据加密）必须经过人工 Review

### 性能基线

- API 响应时间 P95 ≤ **[阈值，如 500ms]**
- 页面首屏加载时间 ≤ **[阈值，如 3s]**（如适用）
- 数据库查询不得出现全表扫描（索引覆盖率要求）

---

## 安全要求

- 所有用户输入必须经过校验和清洗，禁止直接拼接 SQL
- 密码存储必须使用 **[算法，如 bcrypt / argon2]**，禁止明文或 MD5
- API 通信必须使用 HTTPS，禁止 HTTP
- 敏感数据（Token、密钥）禁止提交到版本控制，统一使用环境变量
- 依赖项必须定期扫描安全漏洞（**[工具，如 Dependabot / Snyk]**）

---

## 架构约束

- 模块间禁止循环依赖
- 数据访问层必须通过统一的抽象层封装，禁止在业务逻辑中直接编写原始查询
- 所有外部服务调用必须有超时配置和重试机制
- 日志级别规范：ERROR（需要立即处理）/ WARN（需要关注）/ INFO（业务流程）/ DEBUG（调试信息）
