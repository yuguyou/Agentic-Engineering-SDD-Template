# Agentic Engineering 软件开发实践指南

> 该指南以 **Agentic Engineering（AE）** 为范式主线，以 **Spec-Driven Development（SDD）** 为配套文档体系，整合为一套完整的 AI 时代软件开发体系：
> - **AE 是范式主线**：定义人机协作的工作方式——如何指挥、编排和验收 AI Agent
> - **SDD 是配套载体**：提供结构化文档体系，为 AE 的执行提供可追溯的规范支撑
>
> 本文从第一性原理出发，推导核心实践，再落地为可操作的文档体系与执行流程。

---

## 一、为什么需要 AE + SDD 体系？

### 1.1 Software 3.0 与 Agentic Engineering 的兴起

#### 软件三代演进

Karpathy 将软件发展划分为三代：

| 时代 | 核心方式 | 编程语言 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **Software 1.0** | 人类编写显式逻辑指令 | C++、Python、Java | 传统应用开发 |
| **Software 2.0** | 神经网络从数据中学习参数 | 数据集 + 训练目标 | Tesla Autopilot、推荐系统 |
| **Software 3.0** | 自然语言驱动 LLM 执行任务 | 自然语言（Prompt） | AI Agent 编写和维护代码 |

#### 操作系统类比

在 Software 3.0 时代，Karpathy 提出了一个精确的架构类比来帮助工程师理解新范式：

```
模型权重 = CPU        → 固定的处理引擎（推理能力）
上下文窗口 = RAM      → 有限的工作记忆（当前任务的所有信息）
Prompt = 编程语言     → 通过自然语言"编程"来控制输出
```

这个类比的关键洞察是：**上下文窗口（Context Window）是核心杠杆**。正如程序员通过管理 RAM 中的数据来控制程序行为，Software 3.0 时代的工程师通过管理上下文窗口中的信息来控制 AI 的产出。工程师的角色从"写代码的人"转变为"管理 AI 上下文的架构师"。

### 1.2 从 Vibe Coding 到 Agentic Engineering

#### Vibe Coding 的价值与局限

Vibe Coding 是 Software 3.0 早期的流行实践：开发者用自然语言描述需求，直接接受 AI 输出，凭直觉快速迭代。

**价值**：极大降低了软件开发门槛，原型验证速度快，适合个人项目和小型工具。Karpathy 称之为"抬高了开发底线（raising the floor）"——让更多人能够构建软件。

**局限**：Vibe Coding 的本质是**用速度换取了理解和控制**，无法承载生产级系统的质量要求：
- **无法控制质量**：AI 黑盒生成，没有系统化验收机制
- **不可扩展**：多人协作时 AI 行为不一致，代码风格混乱
- **技术债务快速积累**：凭直觉接受的代码，边界条件和错误处理往往缺失
- **无法用于生产**：对安全性、性能、可维护性无保障

#### Agentic Engineering 的定义

Agentic Engineering 是 Vibe Coding 的专业演进，目标是"突破生产力上限（raising the ceiling）"。核心转变是：

> **不再"让 AI 帮你写代码"，而是"将工程纪律与 AI 能力系统性结合"**

编程单元从"写代码行"变为"委托宏操作（macro actions）"——如"实现这个功能""重构这个子系统""设计这个模块的 API"。工程师扮演架构师、编排者与验收者的角色，AI 是强大但不完美的执行者。

**瓶颈的本质转移**：AI 时代开发者的核心价值不再是编码速度，而是**架构品味（taste）、技术判断力（judgment）和验证能力**。

> "You can outsource your thinking, but you cannot outsource your understanding." —— Karpathy 在多次演讲中引用此名言（原出处为 X 用户 @kache）

| 维度 | Vibe Coding | Agentic Engineering |
| :--- | :--- | :--- |
| **交互模式** | 1 个开发者 vs 1 个 AI 对话框 | 1 个架构师 vs N 个自主 Agent |
| **核心逻辑** | 看到什么 → 说什么 → 跑什么 | 制定规范 → 任务编排 → 闭环验证 |
| **编程单元** | 手写代码行 | 委托宏操作 |
| **代码控制** | AI 黑盒生成，肉眼测试 | 规范驱动生成，自动化验证 |
| **成功关键** | 提示词技巧（Prompting） | 工程化编排（Orchestration） |
| **适用规模** | 原型、MVP、小型工具 | 企业级应用、复杂重构、长周期维护 |

## 二、第一性原理：三条公理

> 本章建立后续所有实践推导的理论地基。三条公理分别定义了人机协作的三个基本要素——**活动**（我们在做什么）、**工具**（我们用什么）、**人**（谁在做）——各自独立，无法互相推导。

### 2.1 公理一：意图转化链与信息损耗

> **软件工程的本质是将人类头脑中模糊的意图，通过一系列信息变换，逐步精确化为机器可执行的代码。这条链的每一步都可能引入信息损耗。**

```
人类意图 → 自然语言需求 → 结构化设计 → 形式化代码 → 可执行程序
```

软件工程的固有挑战——需求误解、设计偏差、实现错误——都是这条链上不同位置的损耗表现。几十年来积累的工程方法论（需求评审、设计文档、代码审查、测试体系），本质上都是在链的各个环节设置**校验点**，捕获和修复损耗。

**关键推论**：
- **源头损耗传播最远**：需求和设计阶段的偏差会在后续每个环节被放大，修复代价呈指数级增长
- **AI 并未消除这条链**：AI 改变了链上各环节的执行方式和损耗率，但链本身始终存在。AI 的概率性本质还引入了新的损耗来源——不是人类的误解，而是 AI 的"似是而非"
- **多层验证的必要性**：链上每一层的"正确"含义不同（意图正确 ≠ 设计正确 ≠ 功能正确 ≠ 性能达标），验证策略必须与链的层次对齐

### 2.2 公理二：LLM 的三个本质特征

> **LLM 是一个基于上下文进行概率性推理的系统，具有三个并列的本质特征。**

#### 特征一：上下文决定性

LLM 的输出严格由其接收到的上下文决定——给它什么信息，它就基于什么推理。对训练数据之外的知识（团队业务逻辑、内部架构决策、编码规范）一无所知——不是"不够聪明"，而是"不知道"。

**推论**：模型能力是给定常量，**上下文质量才是团队真正能控制的提效杠杆**。真正影响 AI 产出的不是代码量，而是代码背后的知识——架构决策原因、模块间的隐含契约、编码规范的细节。

#### 特征二：概率性

LLM 本质上是基于概率的"预测下一个 Token"。同样的输入可能产生不同的输出，这不是缺陷而是架构层面的本质属性。能力边界难以精确预测，输出不可保证正确。

**推论**：

**错误累积效应**——在意图转化链上（公理一），每一步的未校验偏差会成为下一步的输入前提。偏差不会自然消解，而是趋向累积放大：

```
小偏差 → 未校验 → 成为下一步输入 → 偏差放大 → 最终结果严重偏离意图
```

一次性让 AI 完成多步任务，最终结果几乎必然偏离原始意图，且越复杂的任务偏差越严重。

#### 特征三：工作记忆的有限性与易失性

LLM 的上下文窗口既有容量上限，又是易失的：
- **有限**：超长上下文出现"Lost in the Middle"现象——模型对中间位置信息的回忆准确度显著下降
- **易失**：会话结束后，所有讨论结论和决策历史直接归零，没有跨会话的持久记忆
- **压缩有损**：当上下文超出窗口限制时，框架采用 compaction 机制压缩信息，但哪些信息"关键"本身就是一个有损判断

**推论**：没有规范文档，每次新会话 AI 就会重新猜测技术决策。同样的问题在不同会话里可能给出不同答案，导致代码风格混乱，需要大量返工。

#### 三特征的组合效应

三个特征不是孤立的，它们的**组合**产生了最深远的影响：

| 组合 | 效应 | 应对策略 |
| :--- | :--- | :--- |
| 上下文决定性 + 概率性 | 上下文质量决定输出的期望质量上限，但不保证正确性 | 高信噪比上下文 + 验证 |
| 概率性 + 意图转化链 | 多步任务中错误指数级累积 | 小步推进 + 每步校验 |
| 记忆易失 + 上下文决定性 | 跨会话行为不一致，AI 无法自主维持决策连贯性 | 结构化文档作为持久记忆 |
| 记忆有限 + 上下文决定性 | 关键信息被挤出或稀释，长任务后期决策质量下降 | 按需注入 + 渐进式披露 |

### 2.3 公理三：人类认知是稀缺资源

> **工程师的注意力、判断力和决策能力是有限的，是整个人机协作系统的瓶颈。**

这是认知心理学的基本结论——工作记忆容量有限、注意力是零和资源、决策疲劳是真实现象——不因技术进步而改变。

在 AI 时代，制约产出的主要因素不再是编码速度，而是工程师**审查、判断和决策**的能力上限。当 AI 每小时生成数百行代码等待审查时，问题不再是"写不出来"，而是"看不完、想不清"。

**关键推论**：
- **最优策略不是最大化 AI 的产出量，而是最优化工程师认知带宽的分配**——将工程师从机械性工作（查阅 API、编写样板代码、构造测试数据）中解放出来，专注于架构判断和系统设计
- **小变更比大变更容易审查**——面对大型的未经中间审查的变更，审查者需要从头重建上下文，认知成本远高于分步审查
- **AI 引导优于 AI 直接给方案**——让人类在 AI 引导下自己理解问题，比 AI 直接给方案后人类从头重建理解，认知成本更低

### 2.4 三条公理如何指向 AE + SDD

三条公理分别指向两大解法的必要性：

| 公理 | 揭示的核心挑战 | 指向的解法 |
| :--- | :--- | :--- |
| **公理一**（意图转化链） | 源头损耗传播最远，多步偏差累积 | **AE**：多层验证 + AI 全链条参与 |
| **公理二**（LLM 特征） | 输出由上下文决定，概率性不可消除，记忆易失 | **SDD**：规范文档锚定上下文 + **AE**：小步验证控制偏差 |
| **公理三**（认知稀缺） | 人类审查能力是瓶颈，不能无限增加 AI 产出量 | **AE**：最优化认知分配 + 分步审查 |

两者的互补关系：

```
SDD（地基）                    AE（工作方式）
─────────────────────          ─────────────────────
规范文档锚定"做什么"      +      Context Engineering 给 AI 精准上下文
文档约束 AI 行为边界      +      人机分工 决定哪些交给 AI
代码可追溯到规范条目      +      闭环验证 确保每步输出可靠
```

> **一句话**：SDD 解决"AI 不知道该做什么"的问题，AE 解决"AI 知道了但怎么确保做对"的问题。

## 三、核心实践

> 以下六条实践从三条公理推导而出，形成递进结构：**实践 1** 构建上下文的产出与加载机制；**实践 2** 基于知识不对称定义人机分工；**实践 3** 将 AI 的介入从编码扩展到全链条；**实践 4** 通过小任务和多层验证控制错误累积；**实践 5** 用工程化方式治理团队知识；**实践 6** 从错误中提取经验，形成反馈闭环。

### 3.1 上下文工程（Context Engineering）

**推导依据**：公理二（上下文决定性 + 记忆有限性）构成核心矛盾——复杂系统需要更多上下文来保证质量，但 AI 能消费的上下文量是有限的。

Context Engineering 是解决这一矛盾的系统性方法。Karpathy 将其通俗定义为"填充上下文窗口以获得下一步所需恰当信息的精妙艺术与科学"；Anthropic 在其工程博客中进一步将其系统化为：

> **策划和维护 LLM 推理过程中最优 Token 集合的策略**——涵盖系统指令、工具、外部数据、消息历史等所有可能进入上下文窗口的信息。其指导原则是：**寻找能够最大化产生期望结果可能性的最小、高信号 Token 集。**

与 Prompt Engineering 仅关注单次指令编写不同，Context Engineering 关注的是多轮推理和长时间跨度中**整个上下文状态的动态管理**。具体需要解决三个问题：

#### 3.1.1 结构化持久化：将隐性知识显式化为文档

真正影响 AI 输出质量的往往不是代码本身，而是代码背后的知识——架构决策的原因（"为什么选择事件驱动而非同步调用"）、模块间的隐含契约（"服务 A 必须在服务 B 之前初始化"）、编码规范的微妙之处（"所有数据库操作必须通过 Repository 层"）。这些知识之所以不在代码中，是因为代码表达的是"做什么"，而非"为什么这样做"。

**SDD 的解法**：用 spec.md、architecture.md、AGENTS.md 等结构化文档，将隐性知识显式化为 AI 可消费的持久记忆。文档是持久的，AI 每次会话重新加载文档，行为就可以复现和预测。

#### 3.1.2 Docs as Code：文档与代码同仓库

完成的规范文档作为代码保存在仓库中，与源代码共同版本化：
- **可检索**：AI 需要某个模块的上下文时，直接从仓库检索对应文档，而非依赖工程师临时口述
- **可追溯**：文档变更有 Git 历史，与代码变更一一对应
- **防过期**：过期的文档比没有文档更危险——它会注入错误的上下文。同仓库、同分支、同 MR 流程天然保持同步

#### 3.1.3 渐进式注入：按需加载，避免上下文腐化

随着上下文窗口中 Token 数量增加，模型准确回忆信息的能力逐步下降——这不是突然的断崖，而是性能梯度的持续衰减（上下文腐化 / Context Rot）。

**解法**：从预先加载所有数据，转向"即时"上下文策略——按需注入当前任务所需的上下文，而非一次性全量灌入：

| 上下文类型 | 加载策略 | 示例 |
| :--- | :--- | :--- |
| **Rules**（全局约束） | 会话开始时预加载 | AGENTS.md、constitution.md |
| **Spec**（任务规范） | 按当前任务按需引用 | spec.md 的具体条目 |
| **Skills**（领域知识） | 被触发时才加载 | 编码规范、设计原则 |
| **外部数据** | 工具调用时动态获取 | API 文档、数据库 Schema |

目标不是把所有信息塞给 AI，而是提供**最小化、高信噪比的上下文集合**：给对的知识，在对的时机，以对的形式。

### 3.2 基于知识不对称的人机分工（乔哈里窗模型）

**推导依据**：公理二（上下文决定性）定义了 AI 的知识边界——通用知识丰富，私有知识为零；公理三定义了人类的能力边界——认知资源有限，不可能在所有领域都是专家。

两个边界的**交叉**产生四种知识状态，每种状态对应不同的最优协作策略：

| | **AI 知道**（通用知识） | **AI 不知道**（私有知识） |
| :--- | :--- | :--- |
| **人类知道** | **开放区**：极致自动化 | **盲区**：上下文注入 |
| **人类不知道** | **潜能区**：知识杠杆 | **未知区**：协同探索 |

#### 开放区：人类知道 + AI 知道 → 极致自动化

意图明确且属通用知识——AI 拥有全部信息，人类介入只增加延迟。

**策略**：直接下达指令，最大化自动化。
**场景**：样板代码生成、单元测试编写、格式化重构、标准 CRUD 实现。

#### 盲区：人类知道 + AI 不知道 → 上下文注入

任务涉及团队私有知识，AI 无从得知。不显式注入，AI 只能基于通用知识猜测——这是"AI 生成的代码不符合项目规范"的根源。

**策略**：通过 spec、Rules、Skills 等载体注入私有上下文。
**场景**：涉及内部架构约束、特定编码规范、业务逻辑的任务。

#### 潜能区：人类不知道 + AI 知道 → 知识杠杆

人类在某领域缺乏专业知识，但 AI 训练数据恰好覆盖。

**策略**：利用 AI 的广博知识补全人类短板，拓展能力边界。
**场景**：接手陌生技术栈、排查不熟悉的编译器报错、学习新框架的最佳实践。

#### 未知区：双方都不确定 → 协同探索

没有任何一方能独立给出答案。

**策略**：迭代探索——人类提供假设和方向判断，AI 提供验证思路和方案生成，双方在循环中逐步逼近答案。
**场景**：复杂根因分析、全新架构设计、技术选型中的未知权衡。

**分工的核心原则**：识别当前任务落在哪个象限，采用对应的协作策略。一个简单的启发式——问自己两个问题：(1) 这个任务是否涉及团队私有知识？(2) 我自己是否清楚该怎么做？两个答案的组合直接映射到四个象限。

### 3.3 AI 全链条参与：引导者 → 协作者 → 执行者

**推导依据**：公理一（源头损耗传播最远）+ 公理三（认知稀缺，早期阶段尤其耗费认知）。

AI 不应仅参与编码阶段。从意图转化链看，越早期的损耗修复代价越高。当 AI 被提供了足够的领域上下文时，参与早期阶段的边际成本较低（主要是对话交互），但收益在下游被持续放大。

将乔哈里窗模型沿意图转化链展开，AI 在不同阶段的角色随知识状态变化而递进：

#### 需求澄清阶段：AI 作为"引导者"

任务主要落在**盲区**（人类有业务意图但未结构化）和**潜能区**（AI 可帮助发现遗漏）。

AI 通过结构化提问，帮助工程师将模糊意图显式化为 spec，并利用通用知识发现人类未曾考虑到的：
- 边界条件和异常路径
- 隐含约束和前置依赖
- 类似系统的设计模式和已知陷阱

**为什么是"引导者"而非"方案提供者"**：人类是后续所有决策的最终审批者——如果自己没有深度理解问题，后续的审查和决策都会变成盲审。让人类在 AI 引导下自己得出结论，才是认知成本最低的路径（公理三）。

#### 方案设计阶段：AI 作为"协作者"

spec 将私有知识显式注入后，任务从盲区移向**开放区**。AI 的角色升级——主动分析方案权衡、提出替代设计、检查与架构约束的一致性、识别可扩展性或性能隐患。人类做最终的设计决策。

#### 编码测试阶段：AI 作为"执行者"

有了 spec 和设计文档作为高质量上下文，任务大部分进入**开放区**。AI 可以承担更大的自主性——编写实现代码、生成测试用例、执行重构任务。

**正向循环**：每个阶段产生的结构化文档自然成为下一阶段的高质量上下文——需求澄清产出 spec，spec 指导设计，设计文档指导编码。Context Engineering 的要求被逐步满足，而非一次性解决。

### 3.4 小任务推进 + 多层验证

**推导依据**：公理二（概率性 → 错误累积 + 记忆有限 → 上下文退化）+ 公理三（大变更审查的认知成本远高于分步审查）。

#### 小任务推进：控制错误累积

**必须将大任务拆解为可独立验证的子任务，每步由 AI 执行、人类校验，通过后再进入下一步。** 这同时控制了三个风险：

| 风险 | 控制机制 |
| :--- | :--- |
| 错误累积 | 每步校验，防止偏差放大 |
| 上下文退化 | 为每个子任务构建聚焦的上下文，而非在膨胀的会话中持续追加 |
| 验证成本 | 小变更比大变更容易审查（公理三） |

**任务粒度**：以 30min-2h 为宜（S/M/L）。约束越密集的任务（涉及并发、权限、安全），步长应越短；约束较少的任务（样板代码、格式化），可以给予 AI 更大的自主空间。

#### 多层验证：与意图转化链对齐

公理一告诉我们，意图在链上经过多次变换，每一层的"正确"含义不同。验证策略必须与链的层次对齐，不能只靠测试：

| 验证层次 | 验证什么 | 手段 |
| :--- | :--- | :--- |
| **意图层** | 是否在做对的事 | 需求评审、spec 对照 |
| **实现层** | 代码是否符合规范 | Code Review、规范合规检查 |
| **行为层** | 功能是否正确 | 自动化测试（单元 / 集成 / E2E） |
| **系统层** | 性能/安全是否达标 | 压测、安全审查 |

**测试是必要条件，但不是充分条件**。"通过测试"只覆盖了行为层——它无法捕获"功能正确但架构冲突"的设计层偏差，也无法发现命名规范、并发模型等实现层缺陷。

### 3.5 知识治理（Knowledge as Code）

**推导依据**：公理三（认知稀缺 → 团队成员专长分布不均）+ 公理二（上下文决定性 → AI 只能利用它能"看到"的知识）。

团队最有价值的共有知识——最佳实践、设计原则、编码规范——往往锁在少数资深工程师的脑中。在传统模式下，知识传递依赖 Code Review 中的言传身教，效率低且不可规模化。

**解法**：将团队共有知识像代码一样管理——编码为结构化的 Skill 和 Rules，存入仓库，版本化、可 Review、可迭代：

- **知识的公有化**：当资深工程师的设计原则被编码为 Skill 后，AI 成为知识分发的载体——每个团队成员在与 AI 协作时，都自动获得了团队最佳实践的加持，很大程度上拉平了团队内部的能力差异
- **知识的可演进性**：Skill 和 Rules 与代码同仓库、同流程——通过 MR 提出修改，通过 Review 达成共识，通过版本历史追溯演进

**Skill 与 Rules 的分工**：

| 载体 | 加载方式 | 适用内容 | 体量要求 |
| :--- | :--- | :--- | :--- |
| **Rules** | 全量预加载（每次会话开始） | 极轻量的全局约束 | 严格控制，几十行 |
| **Skills** | 按需触发加载 | 编码规范、架构约束、领域知识 | 可以较大，仅在相关任务时消耗上下文 |

**三层渐进式披露**：大量 Skills 不会拖垮上下文窗口：

| 层级 | 内容 | 加载时机 | Token 成本 |
| :--- | :--- | :--- | :--- |
| **L1: Metadata** | YAML frontmatter（名称、描述） | Agent 启动时 | ~100/Skill |
| **L2: Instructions** | SKILL.md 主体指令 | Skill 被触发时 | < 5k |
| **L3: Resources** | 参考文档、脚本、静态资源 | 被显式引用时 | 按需 |

### 3.6 错误驱动的反馈闭环（Error-Driven Refinement）

**推导依据**：公理二（记忆易失 → 会话 A 中被纠正的错误，在会话 B 中会以相同概率再次发生）+ 公理一（意图转化链上的损耗很多是模式化、可重复的）+ 公理三（反复纠正同类错误是认知资源的浪费）。

实践 3.5 是**自上而下**的知识供给——编码已有最佳实践。但很多关键知识在项目开始时并不存在，它们是在开发过程中通过犯错和纠错逐步浮现的。

**必须建立从错误到持久化规则的反馈机制**，通过两种互补途径触发：

#### 自动触发：AI 被纠正时主动提炼

当 AI 在协作过程中被用户纠正时，自动评估错误根因，搜索现有 Rules 和 Skills 中是否已有针对该类问题的处理规则。如果没有，建议创建或更新对应的 Rule/Skill，经用户确认后执行。使知识积累成为开发流程的自然副产物。

#### 手动触发：会话结束后集中沉淀

用户主动发起反思，AI 回顾当前对话历史，识别被纠正的错误模式，诊断根因，提出具体的 Rule/Skill 修改建议。适用于一轮密集协作后集中性地沉淀积累的经验。

**核心闭环**：犯错 → 诊断根因 → 检索现有知识 → 创建或更新 Rule/Skill → 预防复发。

> 实践 3.5 自上而下编码已有知识，实践 3.6 自下而上从错误中生长新知识——共同构成完整的知识生命周期。

## 四、SDD 文档体系

> 用结构化文档约束 AI 行为，为所有实现提供可追溯的事实来源。本章将核心实践落地为具体的文档体系。
> 
> **提示**：如需查阅全部文档的速查参考，请参见 [agentic_engineering_files.md](agentic_engineering_files.md)。

### 4.1 五阶段总览与任务复杂度分级

#### 五阶段文档流程

```
Rules → Specify → Plan → Tasks → Execute
  ↓        ↓         ↓       ↓        ↓
约束行为   定义做什么   规划怎么做  拆解行动   执行验收
```

| 阶段 | 核心文档 | 回答的问题 |
| :--- | :--- | :--- |
| **Rules** | AGENTS.md、constitution.md、README.md | AI 如何行动？有哪些不可逾越的约束？项目全貌是什么？ |
| **Specify** | spec.md（+ ui-spec.md） | 要构建什么？如何验收？ |
| **Plan** | architecture.md、data-model.md、contracts/、plan.md、decisions.md、roadmap.md | 用什么技术？怎么构建？先做什么？ |
| **Tasks** | tasks.md、checklist.md、seed-data/（按需） | 原子任务是什么？质量是否达标？ |
| **Execute** | progress.md、changelog.md、proposal.md（MVP 后） | 当前状态？交付了什么？如何管理变更？ |

#### 任务复杂度分级：流程严格程度应与任务风险成正比

**不是所有任务都需要走完整的五阶段流程。** 简单的 bug fix 不需要写完整 spec，但复杂的跨模块重构必须走完整链路。先评估任务复杂度，再决定流程深度：

| 复杂度 | 特征 | 必须的文档 | 可跳过的文档 |
| :--- | :--- | :--- | :--- |
| **简单**（S） | 单文件、局部修改、意图明确 | AGENTS.md（已有） | spec、architecture、plan |
| **中等**（M） | 多文件、单模块、有明确边界 | AGENTS.md + spec.md + tasks.md | architecture、data-model |
| **复杂**（L） | 多模块、跨服务、涉及设计决策 | 完整五阶段文档 | 无 |
| **关键**（XL） | 架构级变更、安全相关、数据迁移 | 完整五阶段 + decisions.md + 安全审查 | 无 |

**判断启发式**：问自己三个问题——(1) 涉及几个文件/模块？(2) 是否需要设计决策？(3) 出错的修复成本有多高？答案越大，流程越完整。

### 4.2 Rules 阶段：设定 AI 行为约束

> 在任何开发工作开始之前，先为 AI 和团队设定不可变的约束和上下文。

#### AGENTS.md — AI 编码行为契约

固化 AI 跨会话的编码行为。文件名因工具而异：Claude Code 用 `CLAUDE.md`，Cursor 用 `.cursorrules`，通用标准为 `AGENTS.md`。建议控制在 300 行以内。

**核心模板**：

```markdown
## 编码原则
- 先思考再编码：理解需求后，输出实现方案，等待确认再编码
- 精准修改：只修改任务要求的代码，不"顺便"改其他地方
- 定义可验证目标：每次实现结束后给出验证步骤

## 禁止行为
- 禁止在未明确要求时引入新依赖
- 禁止删除现有测试
- 禁止修改数据库 Schema 而不同步更新 data-model.md

## 技术栈约束
- 编程语言：[语言及版本]
- 框架：[框架及版本]
- 测试框架：[框架]

## 项目结构
[简要描述目录结构和各模块职责]
```

#### constitution.md — 最高技术约束与质量红线

项目的"宪法"，约束所有技术决策，防止架构漂移：

```markdown
## 技术约束
- 编程语言：仅使用 [语言及版本]
- 数据库：仅使用 [数据库]
- 部署环境：[环境描述]

## 质量红线
- 测试覆盖率必须 ≥ [阈值]%
- 任何 PR 必须通过所有自动化测试才能合并
- 安全相关代码必须经过人工 Review

## 不可变约束
- [列出绝不能违反的技术/安全约束]
```

#### README.md — 项目入口文档

为 AI 和新成员提供项目高层上下文，是所有人进入项目的第一个文档：
- 项目背景与目标
- 技术栈概览
- 环境搭建步骤（须可一键执行）
- 快速启动指南
- 核心文档索引

### 4.3 Specify 阶段：定义做什么

#### spec.md — 唯一真理来源

所有实现必须可追溯至此文档的具体条目。AI 生成代码时引用条目编号（如"实现 F-001"），保持可追溯性。任何"代码里有但 spec 里没有"的功能，都是未经授权的范围蔓延。

**核心模板**：

```markdown
# [项目名称] 规范文档

## 项目概述
- 目标：[一句话描述项目要解决的问题]
- 核心用户：[目标用户群]
- 成功指标：[可量化的成功标准]

## 功能清单
| 编号 | 功能 | 优先级 | 状态 |
| :--- | :--- | :--- | :--- |
| F-001 | 用户注册 | P0 | 草稿 |
| F-002 | 邮件验证 | P0 | 草稿 |

## 功能详情

### F-001 用户注册
**描述**：[功能的详细描述]

**验收标准**：
- **AC-001-01**：
  - Given：用户在注册页面，且邮箱未被注册
  - When：用户填写有效邮箱和密码，点击注册
  - Then：账户创建成功，验证邮件在 60 秒内到达
- **AC-001-02**：
  - Given：用户在注册页面，且邮箱已被注册
  - When：用户填写该邮箱，点击注册
  - Then：显示"该邮箱已注册"错误提示

## Out of Scope
- [明确列出本迭代不做的功能，防止 AI 自行实现]
```

#### ui-spec.md — UI 实现契约（按需）

当项目涉及前端界面时，补充 UI 规范：页面布局、交互流程、设计系统约束。

### 4.4 Plan 阶段：规划怎么做

文档编写顺序：`spec.md` → `architecture.md` → `data-model.md` + `contracts/` → `plan.md`

#### architecture.md — 系统骨架蓝图

```markdown
## 系统架构图
[C4 模型或模块图]

## 模块划分与职责
| 模块 | 职责 | 对外接口 |
| :--- | :--- | :--- |

## 通信方式
- 同步调用：[场景及协议]
- 异步消息：[场景及消息队列]
- 事件驱动：[场景及事件总线]

## 关键约束
- [性能、安全、可用性等约束]
```

#### data-model.md — 数据持久化权威模型

ER 图、表结构与字段定义（含类型、约束、默认值）、索引策略。**所有数据库 Schema 变更必须先更新此文档**。

#### contracts/ — 服务间 API 硬契约

OpenAPI / GraphQL Schema，按服务分文件。**接口变更须先更新契约文件，再修改实现**。

#### plan.md — 技术实施路径

技术栈细化（含版本）、模块实现顺序、第三方集成方案、错误处理策略。聚焦"怎么实现"，不重复 architecture 已有的架构决策；API 接口设计由 `contracts/` 承担，此处只描述实现思路。

#### decisions.md — 架构决策记录（ADR）

保存"为什么这样做"的决策记忆库，避免重复讨论，支持未来回溯。**只追加不删除**（可标注"已废弃"）；应在决策当时记录，避免事后补写导致细节丢失：

```markdown
## ADR-001: 选择 PostgreSQL 而非 MongoDB
- 日期：2026-05-19
- 状态：已采纳
- 背景：[为什么需要做这个决策]
- 决策：[最终选择及原因]
- 备选方案：[被否决的方案及否决原因]
- 影响：[对架构和开发的影响]
```

#### roadmap.md — 交付路线图

在里程碑层面规划产品交付节奏，将 spec 中的功能集合拆解为可迭代的阶段目标：
- 里程碑划分（含目标与时间）
- MVP 范围定义
- 各迭代周期的交付目标与优先级排序
- 依赖关系说明

里程碑粒度不宜过细，原子任务由 `tasks.md` 承担。

### 4.5 Tasks 阶段：拆解行动

#### tasks.md — 原子化行动清单

每个任务包含明确的描述、验收命令、复杂度和依赖关系：

```markdown
## 当前迭代任务

- [ ] T-001：实现用户注册 API（关联 F-001，AC-001-01~05）
  - 验收：`pytest tests/test_auth.py::test_register -v` 全部通过
  - 复杂度：M | 依赖：无
- [ ] T-002：实现邮件验证发送（关联 F-002）
  - 验收：运行脚本，收到测试邮件
  - 复杂度：S | 依赖：T-001
- [ ] T-003：实现登录 API（关联 F-003，AC-003-01~03）
  - 验收：`pytest tests/test_auth.py::test_login -v` 全部通过
  - 复杂度：M | 依赖：T-001

## 任务依赖关系
T-001 → T-002（串行）
T-001 → T-003（串行）
T-002 / T-003（可并行）
```

#### seed-data/（按需）— 测试基础数据

为集成测试与 E2E 测试提供可运行的基础数据。采用结构化格式（JSON / SQL / YAML / CSV），文件命名应体现场景含义（如 `user-normal.json`、`order-edge-cases.sql`）。有集成测试或 E2E 测试时必需，在编写 tasks 前准备好。

#### checklist.md — 发布前质量关卡

聚焦跨任务的系统级验收，与 `spec.md` / `tasks.md` 的单功能验收形成互补。每项须有明确的通过/失败判断标准：

```markdown
## 发布前检查项
- [ ] 所有 tasks.md 中的任务已完成并勾选
- [ ] 所有自动化测试通过
- [ ] Code Review 已完成
- [ ] spec.md 中的所有验收标准已验证
- [ ] 无"代码里有但 spec 里没有"的功能
- [ ] decisions.md 已记录本迭代的重要决策
- [ ] 文档已同步更新
```

### 4.6 Execute 阶段：执行追踪与变更管理

#### progress.md — 在途状态仪表盘

各任务当前状态、遗留问题与负责人、测试结果摘要、阻塞原因及预计解除时间。任务状态枚举：待做 / 进行中 / 完成 / 阻塞。每条 task 完成后立即更新。

#### changelog.md — 已交付成果的历史档案

按版本组织的累积交付记录。遵循 [Keep a Changelog](https://keepachangelog.com) 格式，按时间倒序追加，历史记录不得删改。

#### proposal.md（MVP 后）— 变更管理

MVP 发布后出现新需求时创建，管理功能演进，保证每次需求增/删/改都有记录与评审：
- 变更动机与背景
- 影响范围分析（波及的 spec 条目、API 接口、数据模型）
- 实施方案
- 评审状态（草稿 / 通过 / 拒绝）

评审通过后按标准流程触发级联更新：`proposal -> 修改 spec -> 重新 plan -> 重新 tasks`。

## 五、执行循环

> SDD 定义"做什么"，AE 定义"怎么做"，执行循环是两者的交汇点——工程师每天执行的核心工作流程。

### 5.1 单任务标准执行流程

每个 Task 遵循以下标准循环：

```
┌─────────────────────────────────────────────────────┐
│               单任务执行循环                           │
│                                                     │
│  1. 读取 tasks.md，选取下一条任务                      │
│     ↓                                               │
│  2. 构建上下文：加载相关 spec 条目 + 设计文档            │
│     ↓                                               │
│  3. 下达结构化指令给 AI                                │
│     ↓                                               │
│  4. AI 生成代码                                      │
│     ↓                                               │
│  5. 运行验收命令                                      │
│     ├─ 通过 → 6a. 勾选 tasks.md，更新 progress.md     │
│     │         → 返回步骤 1                            │
│     └─ 不通过 → 6b. 分析原因                          │
│                 ├─ 规范问题 → 更新 spec/plan，         │
│                 │             记录 decisions.md       │
│                 └─ 实现问题 → 重新指令 AI               │
│                               (先反思自己的上下文)      │
└─────────────────────────────────────────────────────┘
```

**关键规则**：
- **AI 每次只执行一条 task**，不允许批量执行
- 验证不通过时，**先反思是否是自己的指令、编排或上下文出了问题**——AI 的输出质量是工程师上下文管理能力的映射
- 如果 AI 反复犯同类错误，触发实践 3.6 的反馈闭环——将错误模式沉淀为 Rule/Skill

### 5.2 AI 指令的结构化写法

结构化指令是 Context Engineering 的核心实践——给 AI 精准、可追溯的上下文，而非模糊的口头描述：

```
实现 tasks.md 中的 T-001：用户注册 API 端点。

参考：
- 功能规范：spec.md F-001（验收标准 AC-001-01 ~ AC-001-05）
- API 定义：contracts/user-api.yaml POST /auth/register
- 数据模型：data-model.md users 表
- 实现方案：plan.md 3.1 用户认证模块

约束：
- 遵循 AGENTS.md 中的编码规范
- 密码必须使用 bcrypt 加密（constitution.md 安全约束）

完成后给出可运行的验收步骤。
```

**指令的核心要素**：
1. **明确任务**：引用 tasks.md 中的任务编号
2. **提供上下文**：引用 spec 条目、API 契约、数据模型
3. **声明约束**：引用 AGENTS.md 和 constitution.md 中的约束
4. **要求验收步骤**：让 AI 给出可运行的验证方法

### 5.3 任务粒度与拆解原则

#### 粒度标准：30min-2h，按约束密度调整

| 复杂度 | 时间估计 | 特征 | 示例 |
| :--- | :--- | :--- | :--- |
| **S**（简单） | 30min 以内 | 单文件、明确逻辑 | 添加一个工具函数、修复格式问题 |
| **M**（中等） | 30min - 1h | 多文件、清晰边界 | 实现一个 API 端点、添加一个组件 |
| **L**（复杂） | 1h - 2h | 跨模块、涉及设计 | 实现完整认证流程、重构数据访问层 |

**超过 2h 的任务必须继续拆分**——这不是建议，而是规则。

#### 任务依赖关系

- **串行任务**：上游完成并验证后才能启动下游（如：数据模型 → API → UI）
- **并行任务**：无依赖关系的任务可同时分配给不同 Agent 并行执行
- 在 tasks.md 中明确标注依赖关系，避免 AI 跳过前置依赖

### 5.4 会话上下文管理

#### 会话开始：加载哪些文档

```
必须加载（每次会话）：
├── AGENTS.md          → AI 行为约束
├── constitution.md    → 技术红线
└── 当前任务的 spec 条目 → 任务上下文

按需加载（与任务相关时）：
├── architecture.md    → 涉及架构的任务
├── data-model.md      → 涉及数据库的任务
├── contracts/         → 涉及 API 的任务
└── decisions.md       → 需要了解历史决策时
```

#### 会话结束：沉淀哪些决策

每次会话结束前，检查是否需要：
- 将重要的技术决策记录到 `decisions.md`
- 更新 `progress.md` 中的任务状态
- 如果 AI 被纠正过，评估是否需要更新 Rules/Skills（实践 3.6）

#### 会话过长：如何切换续接

当会话过长时（经验上超过数十轮对话后），上下文腐化风险显著增加。具体阈值因模型和任务复杂度而异，应当：
1. 将当前进度记录到 `progress.md`
2. 将重要决策记录到 `decisions.md`
3. 开新会话，重新加载上下文文档
4. **不在超长会话中继续**——防止"Lost in the Middle"效应

## 六、变更管理与持续改进

### 6.1 标准变更流程

MVP 发布后有新需求时，**严禁直接修改代码**，必须走标准变更流程：

```
需求变更 → 评审 → 更新 spec.md → 更新 plan（如需） → 重新拆 tasks → 执行
```

**为什么先改 spec 再改代码**（公理二·上下文决定性）：如果直接改代码，AI 在后续会话中读取的 spec 仍然是旧版本，会基于过期的上下文做决策。spec 作为 Single Source of Truth，必须始终反映当前的真实需求。

### 6.2 错误反馈闭环的落地

当 AI 犯错时的标准处理流程：

```
AI 犯错
  ├─ 1. 先反思自己：指令是否清晰？上下文是否充分？约束是否遗漏？
  ├─ 2. 修正指令，让 AI 重新执行
  ├─ 3. 评估是否是系统性问题
  │     ├─ 否 → 继续
  │     └─ 是 → 4. 沉淀为规则
  │           ├─ 编码规范类 → 更新 AGENTS.md 或相关 Skill
  │           ├─ 架构约束类 → 更新 constitution.md
  │           └─ 领域知识类 → 创建新的 Skill
  └─ 5. 验证更新后 AI 行为是否改善
```

## 七、快速上手

### 7.1 新项目启动 Checklist

#### 步骤 1：规则设定（Rules - Day 1 第一件事）

- [ ] 创建 `AGENTS.md`：编码原则 + 禁止行为 + 技术栈约束（5-10 条规则）
- [ ] 创建 `constitution.md`：质量红线 + 不可变约束
- [ ] 创建 `README.md`：项目背景、环境搭建指南

#### 步骤 2：Specify 阶段

- [ ] 与 AI 协作完成 `spec.md`：
  - 项目概述和成功指标
  - 功能清单（编号 + 优先级）
  - 每个 P0 功能的验收标准（Given/When/Then）
  - Out of Scope 列表
- [ ] （如有前端）创建 `ui-spec.md`

#### 步骤 3：Plan 阶段

- [ ] 创建 `architecture.md`：系统架构图 + 模块划分
- [ ] 创建 `data-model.md`：核心表结构
- [ ] （如有 API）创建 `contracts/`：OpenAPI / GraphQL Schema
- [ ] 创建 `plan.md`：技术栈 + 实现思路
- [ ] 创建 `decisions.md`：初始化架构决策记录
- [ ] 创建 `roadmap.md`：划分里程碑与优先级

#### 步骤 4：Tasks 阶段

- [ ] 准备 `seed-data/`（按需）：创建基础测试数据
- [ ] 创建 `tasks.md`：将 plan/spec 拆解为 5-15 个原子任务

- [ ] （MVP 后）创建 `proposal.md`：管理新需求变更

### 7.2 MVP 最小文档集（3-4 个文档即可开始）

对于个人项目或小型 MVP，**不需要完整的五阶段文档**。以下 4 个文档足以启动：

| 文档 | 最小内容 | 为什么必须有 |
| :--- | :--- | :--- |
| **AGENTS.md** | 5-10 条编码规则 | 约束 AI 跨会话的行为一致性 |
| **spec.md** | 功能清单 + 核心功能的验收标准 | 防止 AI 猜测需求，提供可追溯的验收基准 |
| **tasks.md** | 5-10 个原子任务的 Checklist | 控制执行粒度，防止错误累积 |
| **技术方案** | 技术栈 + 数据库结构（可并入 spec） | 约束 AI 的技术选型 |

**可以省略的文档**（MVP 阶段）：
- `README.md`（内部临时项目可省略）
- `constitution.md`（规则并入 AGENTS.md）
- `architecture.md`（小项目架构简单，可口头说明）
- `contracts/`（无多服务时不需要）
- `decisions.md`（小项目决策少，可后补）
- `roadmap.md`（MVP 只有一个里程碑）
- `seed-data/`（初期可硬编码）
- `proposal.md`（MVP 阶段不需要严格的变更管理）

### 7.3 最佳实践与反模式

#### ✅ 应该做的

| 实践 | 说明 |
| :--- | :--- |
| **规范先行** | 任何功能在编码前，必须有对应的 spec 条目和验收标准 |
| **小步快跑** | 每次只让 AI 实现一个 task，完成后立即验收 |
| **结构化指令** | 引用 spec 条目编号，给 AI 提供精准上下文，而非口头描述 |
| **多层验证** | 自动化测试 + Code Review + 架构一致性检查，三者缺一不可 |
| **记录决策** | 重要技术选型决策立即记入 `decisions.md`，防止知识流失 |
| **读懂代码** | 不理解 AI 生成的核心逻辑，不提交、不部署 |
| **犯错先反思自己** | AI 出错时先检查自己的指令和上下文，再归咎于 AI 能力 |
| **沉淀错误经验** | 重复出现的 AI 错误，沉淀为 Rule/Skill 预防复发 |

#### ❌ 不应该做的

| 反模式 | 后果 |
| :--- | :--- |
| **跳过 spec 直接编码** | AI 猜测的功能逻辑几乎必然需要大量返工 |
| **一次让 AI 实现多个 task** | 错误累积，难以定位问题，验收困难 |
| **口头描述需求** | 口头描述随对话丢失，只有文档才是持久的 |
| **"看起来对"就接受** | 必须运行 tasks.md 中写明的验收命令 |
| **需求变更时直接改代码** | 必须先更新 spec，再改代码 |
| **忽略 Out of Scope** | 未明确说"不做"的功能，AI 可能自行实现 |
| **在超长会话中继续工作** | 上下文腐化导致后期决策质量严重下降 |
| **把所有代码塞给 AI** | 信噪比急剧下降，关键信息被淹没 |

## 八、范式总结

Agentic Engineering 的本质是**工程纪律与 AI 能力的系统性结合**——在保持甚至提升质量标准的前提下，大幅提升研发效能：

| 核心要素 | 含义 |
| :--- | :--- |
| **规范是产品** | Spec 是唯一事实来源，代码是可再生的衍生物 |
| **上下文即能力** | 为 AI 提供高质量、高信噪比的上下文，是团队真正能控制的提效杠杆 |
| **人类是架构师** | AI 是执行者，人类负责架构决策、品味判断和最终质量 |
| **任务是单元** | 原子化任务 + 闭环验证，是控制 AI 错误累积的根本手段 |
| **文档即记忆** | 文档与代码同仓库（Docs as Code），团队知识持续沉淀，不随对话消散 |
| **知识在生长** | 自上而下编码最佳实践 + 自下而上从错误中提炼，知识持续演进 |

> **核心哲学**：你可以委托 AI 执行，但不能委托 AI 理解。SDD 确保"做正确的事"，AE 确保"正确地做事"。
